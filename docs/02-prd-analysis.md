# 02. 요구사항 분석 및 PRD

## 1. 분석 개요

본 문서는 냉장고 레시피 추천 앱의 PRD를 구현 가능하고 테스트 가능하도록 분석한 결과입니다.
누락된 질문, 미정의 가드레일, 스코프 리스크, 미검증 가정, 누락된 수락 기준, 엣지 케이스를 식별합니다.

## 2. 누락된 질문 (Missing Questions)

1. **영수증 OCR의 대상 마트/포맷 범위는?** - 한국 마트(이마트, 홈플러스, 쿠팡 등)마다 영수증 포맷이 다름. 학습 데이터셋 범위와 정확도 기준을 정할 수 없음.

2. **레시피 데이터의 소스는?** - 자체 DB 구축인지, 외부 API(만개의 레시피 등) 연동인지, 크롤링인지에 따라 저작권 이슈, 데이터 품질, 초기 레시피 수량이 달라짐. MVP 출시 시 최소 몇 개의 레시피가 필요한지 정의 필요.

> ✅ **결정됨**: 수동 입력 + 공공 레시피 API(농촌진흥청 등)로 MVP 1,000개 확보, 크롤러 기반 확장은 Post-MVP (섹션 11 참조)

3. **AI 모델 호스팅 전략은?** - 클라우드 AI API(Clova OCR, OpenAI) 사용 시 비용 구조와 응답 시간. MVP에서 AI 추론 비용 예산은?

4. **식재료 인식 정확도의 허용 기준은?** - 사진으로 "당근"을 인식할 때 80% 정확도면 출시 가능한지, 95%가 필요한지. 오인식 시 사용자 경험에 직접 영향.

5. **소비기한 추적의 데이터 입력 방식은?** - 사용자가 직접 날짜를 입력하는지, 영수증에서 자동 추출하는지, 식재료별 평균 소비기한을 자동 부여하는지. 한국 마트 영수증에는 소비기한이 표시되지 않는 경우가 대부분.

6. **오프라인 사용 시나리오는?** - 냉장고 앞에서 앱을 사용할 때 네트워크가 불안정할 수 있음. 식재료 목록과 레시피가 오프라인에서도 접근 가능해야 하는지.

7. **다중 사용자/가구 지원은?** - 한 냉장고를 여러 가구 구성원이 공유하는 경우 어떻게 처리하는지. 가족 간 식재료 공유 기능이 MVP에 필요한지.

8. **알레르기/식이 제한 정보의 신뢰성 보장은?** - 레시피에 알레르기 유발 식재료가 포함된 경우 법적/안전 책임은? 면책 조항이 필요한지.

9. **푸시 알림 인프라는?** - 소비기한 알림을 위해 FCM/APNs 통합 필요.

10. **사용자 인증 방식의 구체적 범위는?** - 소셜 로그인(카카오, 네이버, Google, Apple) 포함 범위. 한국 서비스라면 카카오/네이버 로그인이 사실상 필수.

11. **레시피 추천 알고리즘의 동작 방식은?** - 단순 재료 매칭인지, AI 기반 추천(유사 재료 대체 제안 포함)인지. 매칭 임계값 필요.

12. **수익 모델은?** - 무료+광고 / 프리미엄 구독 / 유료 중 어떤 모델인지에 따라 API 호출 제한, 기능 제한 등 가드레일 설계에 영향.

## 4. 스코프 리스크 (Scope Risks)

### 4.1 AI 기능 3개 동시 개발

- **리스크**: OCR + 사진 인식 + 레시피 추천을 3-4개월 MVP에 모두 포함하면 과도함
- **방지**: 수동 입력을 P0 핵심으로, 영수증 OCR을 P0, 사진 인식을 P1으로 분리

### 4.2 3 플랫폼 동시 출시

- **리스크**: iOS + Android + Web을 모두 KMP로 구현하는 것은 비현실적
- **방지**: MVP는 모바일(iOS + Android)에 집중, Web은 Post-MVP (Next.js 별도)

### 4.3 소셜/커뮤니티 기능의 조기 확장

- **리스크**: 레시피 공유/평점이 데이터 모델에 영향
- **방지**: 데이터 모델 설계 시 확장 고려하되, UI/API는 MVP에서 완전 제외

### 4.4 영수증 OCR의 범위 확대

- **리스크**: 한국어 OCR + NLP + 마트별 포맷 차이 → 별도 프로젝트 규모
- **방지**: MVP에서는 Clova OCR Receipt 모델 활용, 미매칭 항목은 사용자 수동 확인

### 4.5 다국어 지원

- **리스크**: 식재료명의 한영 매핑은 비자명함 ("깻잎" ↔ "Perilla leaf")
- **방지**: MVP는 한국어 전용, 영어는 P2

| 6 | 3-4개월이면 MVP가 가능하다 | 팀 구성과 역량 명시 후, 유사 프로젝트 경험 기반 일정 산정 |

## 6. 누락된 수락 기준 (Missing Acceptance Criteria)

### 6.1 식재료 수동 등록

- 중복 식재료 등록 시 동작 (수량 합산? 별도 항목?)
- 식재료명 자동완성/표준화 ("달걀"과 "계란" 동의어 처리)
- 수량/단위 입력 형식 (자유 텍스트? 드롭다운?)

### 6.2 레시피 추천

- 매칭 로직: 100% 매칭만? 부분 매칭 시 부족 재료 표시?
- 매칭률 기준 (70%? 80%?)
- 추천 결과가 0건일 때의 UX (대체 레시피 제안? 빈 상태 메시지?)

### 6.3 사진 기반 식재료 인식

- 인식 실패 시 사용자 알림 방식
- 부분 인식(5개 중 3개만 인식) 시 UX
- 인식 결과 사용자 확인/수정 플로우 필수
- 처리 시간 상한 (5초? 10초? 30초?)

### 6.4 사용자 인증

- 세션 만료 시간 및 토큰 갱신 방식
- 비밀번호 찾기 플로우 (소셜 로그인 전용이면 불필요)
- 계정 삭제 플로우 (개인정보보호법 의무)

### 6.5 소비기한 알림

- 알림 채널 (푸시? 인앱? 이메일?)
- 알림 빈도 제한 (같은 식재료에 대해 하루에 몇 번?)
- 만료된 식재료 자동 처리 (목록에서 제거? 표시만?)

### 6.6 레시피 상세 화면

- 조리 단계별 타이머 기능 포함 여부
- 인분 조절 시 재료 자동 재계산 여부
- 영양 정보의 출처와 정확도 면책

### 6.7 쇼핑 리스트

- 부족 재료 자동 추가의 트리거 (사용자 명시적 버튼 vs 레시피 선택 시 자동)
- 구매 완료 체크 시 냉장고 재고에 자동 반영 여부

### 6.8 영수증 OCR 스캔

- Clova OCR API 응답 시간 5초 이내 (네트워크 제외)
- 한국 주요 마트(이마트, 홈플러스, 롯데마트, GS25, CU) 영수증 상품명 인식률 80% 이상
- 비식재료 항목(세제, 휴지 등) 자동 필터링 정확도 90% 이상
- 미인식/미매칭 항목은 사용자 수동 확인 플로우로 전환
- 영수증 이미지 해상도 최소 요구: 300dpi 이상 또는 1024x768px 이상
- 처리 실패 시 사용자에게 재촬영 또는 수동 입력 대안 안내
- 한 장의 영수증에서 최대 50개 항목 처리

## 7. 엣지 케이스 (Edge Cases)

| # | 시나리오 | 대응 방안 |
|---|---------|----------|
| 1 | 동일 식재료의 다른 표현 ("계란" vs "달걀" vs "에그") | 식재료 마스터 DB에 동의어(alias) 매핑 테이블 |
| 2 | 반만 사용한 식재료 (양파 반 개) | 소수점 수량 허용, 또는 "충분/부족/없음" 3단계 간소화 |
| 3 | 카메라 권한 거부 | 권한 안내 → 설정 이동 유도 → 수동 입력 대안 제시 |
| 4 | 영수증에 비식재료 항목 포함 (세제, 휴지 등) | 식재료 사전과 매칭, 미매칭 항목은 사용자 확인 |
| 5 | 동시에 같은 냉장고를 관리하는 여러 기기 | 낙관적 잠금(optimistic locking), Last-Write-Wins |
| 6 | 대량 식재료 일괄 삭제 (냉장고 정리) | 일괄 선택/삭제 UI |
| 7 | 레시피 재료와 사용자 재고의 단위 불일치 | MVP는 재료 "유무"만 매칭(수량 무시), 단위 변환은 P2 |
| 8 | 네트워크 오류 중 이미지 업로드 | 업로드 재시도 + 클라이언트 측 이미지 압축 |
| 9 | 빈 냉장고 상태 (식재료 0개) | 온보딩 가이드, 인기 레시피 표시, 식재료 추가 유도 CTA |
| 10 | 식재료 사진에 사람 얼굴/부적절한 이미지 | 콘텐츠 모더레이션, 식재료 미검출 시 재촬영 안내 |
| 11 | 소비기한이 없는 식재료 (쌀, 소금, 식용유) | "소비기한 없음" 옵션, 기본 소비기한 자동 설정 |
| 12 | 레시피 추천 결과가 항상 동일 | 랜덤 요소 또는 조리 이력 기반 다양성 보장 |

## 8. 우선순위 권고사항

### Critical (플래닝 차단 요소)

1. **레시피 데이터 소싱 전략 확정** - 자체 DB / 외부 API / 크롤링 / 하이브리드 중 선택. 데이터 모델, API 설계, 일정 산정 모두에 영향.

2. **AI 기능의 MVP 범위 축소** - 수동 입력(P0) + 영수증 OCR(P0, Clova OCR 활용) + 콘텐츠 기반 추천(P0, 키워드 매칭) + 사진 인식(P1, GPT-4o Vision).

3. **플랫폼 범위 확정** - MVP는 모바일(iOS + Android)만. Web은 Post-MVP로 Next.js 별도 구현.

4. **팀 구성 및 역량 명시** - 개발자 수, AI/ML 전문가 유무, KMP 경험 수준 없이는 일정 산정 불가.

### High (설계 품질에 영향)

5. **식재료 정규화 전략** - 동의어, 카테고리, 단위 체계를 데이터 모델 설계 전에 확정.

6. **인증 방식 확정** - 소셜 로그인 범위(카카오/Google/Apple MVP) 확정.

7. **레시피 매칭 알고리즘 정의** - 콘텐츠 기반 필터링 + 스코어링 매칭 방식 확정.

### Medium (UX 품질에 영향)

8. **오프라인 전략 확정** - SQLDelight 로컬 캐싱 범위, 오프라인 편집 후 동기화 방식.

9. **알림 채널 및 빈도 정책** - 푸시 알림 인프라, 사용자 피로 고려한 빈도 제한.

10. **가구/공유 기능의 MVP 제외** - 1인 가구 대상으로 시작, 가족 공유는 P2.

## 9. MVP 기능 우선순위 매트릭스

### P0 (MVP 필수 - 18~20주, 데이터 시딩 방식에 따라 가변)

| 기능 | 예상 공수 | 비고 |
|------|----------|------|
| 회원가입/로그인 (카카오 + Google + **Apple** OAuth + JWT) | 2주 | |
| 식재료 수동 등록 CRUD + 자동완성 검색 | 2주 | 핵심 기능 |
| 영수증 OCR 스캔 (Clova OCR) | 3주 | 자동 입력 |
| 냉장고 현황 대시보드 | 1.5주 | |
| 레시피 추천 (콘텐츠 기반 매칭) | 3주 | 핵심 가치 |
| 레시피 상세 (재료, 조리법, 이미지) | 1주 | |
| 레시피 즐겨찾기 | 0.5주 | 01-arch bookmarks API/테이블 및 03-design 와이어프레임 반영. 용어: UI="즐겨찾기", API="/bookmark", 아이콘=♡ |
| 소비기한 알림 (D-3, D-1 푸시) | 1.5주 | |
| 프로필/설정 (식이 선호, 알림 설정) | 1주 | |
| 오프라인 캐시 (SQLDelight) | 2주 | |
| 식재료 마스터 데이터 시딩 (500종+) | 1주 | 카테고리별 식재료명, 동의어, 기본 소비기한 (아래 상세) |

**마스터 데이터 시딩 전략:**

## 13. 확정된 설계 결정

### 13.1 식재료 등록 방식: 간편 등록 우선

사용자가 "당근이 있다!" 수준으로 간단하게 등록해도 레시피 추천이 동작하도록 설계한다.

**등록 모드:**

| 모드 | 필수 입력 | 선택 입력 | 사용 시나리오 |
|------|----------|----------|-------------|
| **간편 등록** | 재료명만 | - | "당근 있어!" → 탭 한 번으로 등록 |
| **상세 등록** | 재료명 | 수량, 단위, 소비기한, 보관방식, 메모 | 장보기 후 꼼꼼히 기록할 때 |

- **간편 등록 시 기본값**: `quantity=NULL, unit=NULL, expiry_date=NULL(또는 마스터 default_expiry_days 자동 적용), storage_type=FRIDGE`
- **레시피 매칭은 재료 유무(presence) 기반**: 수량/단위 없어도 추천에 지장 없음
- **소비기한 미입력 시**: 마스터 데이터의 `default_expiry_days`로 자동 계산하여 제안하되, 사용자가 건너뛸 수 있음
- **UI 흐름**: 검색 → 재료 선택 → 즉시 등록 완료 (상세 입력은 "더보기" 펼침)

### 13.2 중복 재료 등록 처리

**규칙: `ingredient_id + storage_type + expiry_date` 조합이 같으면 합산, 다르면 별도 항목**

| 시나리오 | 처리 |
|---------|------|
| 같은 재료 + 같은 보관방식 + 같은 소비기한 | 수량 합산 (수량이 NULL이면 그대로 유지) |
| 같은 재료 + 같은 보관방식 + 다른 소비기한 | 별도 항목 |
| 같은 재료 + 다른 보관방식 | 별도 항목 |
| 간편 등록(소비기한 NULL) + 간편 등록(소비기한 NULL) | 1건만 유지 (중복 무시) |
| OCR/사진 일괄 등록 시 기존 재료와 중복 | 사용자에게 "이미 있는 재료" 표시 → 합산/별도/건너뛰기 선택 |

### 13.3 재료명 정규화 전략

**3단계 매칭 알고리즘:**

```
1단계: 정확 매칭 — name = 입력값
2단계: 동의어 매칭 — aliases @> ARRAY[입력값]
3단계: 유사어 매칭 — pg_trgm similarity > 0.3
```

**정규화 규칙:**

| 항목 | 정책 |
|------|------|
| 대표명(name) | 가장 대중적인 이름 사용 (예: "달걀" > "계란") |
| 동의어(aliases) | 모든 변형 포함 (예: `["계란", "egg", "에그", "유정란", "무정란"]`) |
| 자동완성 | name + aliases 모두 검색 대상, 결과는 대표명으로 표시 |
| OCR/Vision 결과 | AI 반환값을 aliases에서 검색 → 매칭 시 자동 매핑 |
| 미매칭 시 | 사용자에게 "가장 비슷한 재료" 3개 제안 + "직접 선택" UI |
| 신규 재료 | 마스터에 없는 재료는 등록 불가 (MVP). 500종+ 마스터로 커버 |

**마스터 데이터 예시:**

```json
{
  "name": "달걀",
  "aliases": ["계란", "egg", "에그", "유정란", "무정란"],
  "category": "난류",
  "default_unit": "개",
  "default_expiry_days": 21
}
```

### 13.4 레시피 매칭 임계값 및 표시 전략

**최소 표시 기준**: 필수 재료(`is_essential=true`) 1개 이상 보유 AND 매칭률 30% 이상

**구간별 세그먼트:**

| 구간 | 매칭률 | 라벨 | UI 처리 |
|------|--------|------|---------|
| A | 80%+ | "바로 요리 가능" | 녹색 배지, 상단 노출 |
| B | 50~79% | "재료 조금 부족" | 주황 배지, 부족 재료 목록 표시 |
| C | 30~49% | "도전해보세요" | 회색 배지, 접힌 상태 (펼쳐서 확인) |
| - | 30% 미만 | 표시 안 함 | 추천 목록에서 제외 |

**추가 규칙:**
- 대체 재료(`substitute_ids`) 보유 시 해당 재료 보유로 간주하여 매칭률 계산
- 0건일 때 UX: "보유 재료로 만들 수 있는 레시피가 없어요" + 인기 레시피 5개 표시 + "재료 추가하기" CTA
- 정렬: 추천 스코어(섹션 4.3 공식) 내림차순

### 13.5 소비기한 만료 후 처리

**상태별 처리:**

| 상태 | 조건 | UI 표시 | 추천 엔진 |
|------|------|---------|----------|
| 안전 | D-4 이상 | 녹색 배지 | 정상 포함 |
| 임박 | D-3 ~ D-2 | 주황 배지 + 푸시 알림 | 가산점 (임박 재료 활용 레시피 우선) |
| 긴급 | D-1 ~ 당일 | 빨간 배지 + 푸시 알림 | 높은 가산점 |
| 만료 | D+1 이후 | 회색 배지 + 취소선 | **추천에서 제외** |

**만료 후 처리 흐름:**

1. 만료일 다음날 → "만료됨" 표시 (자동 삭제 안 함)
2. 냉장고 목록 하단 "만료된 재료" 별도 섹션으로 이동
3. 사용자 액션 선택:
   - "사용 완료" → 삭제
   - "아직 괜찮아요" → 소비기한 +3일 연장 (최대 2회)
   - "버렸어요" → 삭제
4. 만료 후 **7일 경과** + 사용자 무응답(`expired_notified=true`이고 `updated_at`이 만료일 이후 변경 없음) → 자동 삭제 + 인앱 알림

**소비기한 없는 재료** (쌀, 소금, 식용유 등):
- `expiry_date = NULL` 허용 (간편 등록 시에도 동일)
- 만료 알림 없음, 추천 엔진에서 항상 포함
- 마스터 데이터에서 `default_expiry_days = NULL`로 설정
